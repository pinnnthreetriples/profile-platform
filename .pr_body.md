# Stage 3+4 — Payments and Webhooks

## Summary

End-to-end payment flow using BTCPay Server with the Tether USDt Plugin for USDT on TRON (TRC20). Includes invoice creation via Greenfield API, webhook signature verification, idempotent event processing, and server-side payment/profile status updates.

## Scope

- Supabase migrations: `payments` and `payment_events` tables with RLS
- Supabase admin client (service role, server-only)
- BTCPay Greenfield API client (real implementation, USDT-TRON)
- Webhook signature verification (HMAC-SHA256, constant-time)
- Idempotent webhook processing with audit log
- Payment feature module (server, actions, schemas, types, queries, components)
- `/payment` page: real server-side data, replaces placeholder
- `/payment/success` and `/payment/cancel`: UX-only, no payment mutation
- Unit tests: BTCPay client, webhook verifier, webhook processing, payment server
- E2E smoke tests
- Documentation: PAYMENT_FLOW.md, WEBHOOK_FLOW.md, STAGE_3_4_COMPLETE.md

**Not included:** Polygon/Ethereum UI, USDC, refunds, admin panel, UI polish.

## DB Migration Details

File: `supabase/migrations/20260515000001_create_payments.sql`

### payments table
- `id uuid` PK, `user_id` FK auth.users, `profile_id` FK profiles
- `provider='btcpay'`, `provider_invoice_id` (unique), `checkout_url`
- `amount numeric(12,2)`, `currency='USDT'`, `network='tron'`, `payment_method_id='USDT-TRON'`
- `status` enum: pending/processing/paid/failed/expired/cancelled
- `created_at`, `updated_at` (trigger), `expires_at`, `settled_at`

### payment_events table
- Audit log for all BTCPay webhook events
- Idempotency: unique index on `(provider, event_id) WHERE event_id IS NOT NULL`

## RLS Details

| Table | Policy | Rule |
|-------|--------|------|
| payments | Users can read own payments | `auth.uid() = user_id` |
| payment_events | Users can read own payment events | via payments join |

Both tables: `REVOKE ALL FROM anon, authenticated; GRANT SELECT TO authenticated`. No direct INSERT/UPDATE/DELETE for users.

## BTCPay + Tether USDt Plugin

- **Provider:** BTCPay Server (Greenfield API)
- **Plugin:** Tether USDt Plugin (`BTCPayServer.Plugins.USDt` v0.6.0+, min BTCPay v2.3.7)
- **Payment method id:** `USDT-TRON` (TRON network)
- **Reference:** https://github.com/btcpayserver-tether/BTCPayServer.Plugins.USDt

## USDT-TRON Confirmation

MVP supports USDT on TRON only. Payment method id `USDT-TRON` is used in:
- BTCPay invoice creation (`checkout.paymentMethods: ["USDT-TRON"]`)
- DB constraint (`payment_method_id IN ('USDT-TRON')`)
- UI copy ("Pay with USDT TRC20")

## Webhook Verification Strategy

1. Read raw body (`request.text()`) before any parsing
2. Read `BTCPay-Sig` header (format: `sha256=<hmac-sha256-hex>`)
3. Compute HMAC-SHA256 of raw body with `BTCPAY_WEBHOOK_SECRET`
4. Compare using `timingSafeEqual` (constant-time, prevents timing attacks)
5. Reject with 401 if invalid — no further processing
6. Parse JSON only after verification

## Idempotency Notes

- Unique index: `payment_events(provider, event_id) WHERE event_id IS NOT NULL`
- On duplicate `deliveryId`: return 200, skip processing
- All events stored in `payment_events` (audit log), including unknown types
- Unknown event types: stored but do not mutate payment status

## Security Notes

- Webhook signature verified before any JSON parsing or DB mutation
- Raw body used for signature verification
- Success/cancel pages do not mutate payment state
- Payment amount/currency/network server-controlled
- Admin client used only in server-only files
